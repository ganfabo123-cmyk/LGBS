# LGBS 空壳项目与实例项目差异研究报告

## 一、概述

本文档对比 `deepresearchagent(1)`（空壳模板项目）与 `deepresearchagent`（完整实例项目）之间的核心差异，分析两者在架构设计、节点实现、工作流配置等方面的区别。这些差异将指导 LGBS 用户如何将空壳项目填充为可用的 LangGraph 应用。

---

## 二、核心差异总结

### 2.1 工作流架构差异

| 维度 | 空壳模板项目 | 完整实例项目 |
|------|-------------|-------------|
| **节点组织** | 简单边连接，节点直接相连 | 每个 LLM 节点后接 `_response` 工具节点，形成完整的 Tool Calling 闭环 |
| **状态管理** | 直接修改 `state["messages"]` | 新增 `llm_messages` 字段用于隔离 LLM 消息，使用模块专用状态（如 `state["informationretrieval"]`） |
| **路由逻辑** | 简单的 `route_after_*` 函数，TODO 注释占位 | 完整的条件边函数，实现状态提取和分支判断 |
| **模块间传递** | 无模块间状态传递逻辑 | 实现了 `informationretrieval` → `reportgeneration` 的数据传递 |

## 三、文件级差异详解

### 3.1 state.py 差异

**空壳项目：**
```python
class AgentState(TypedDict):
    topic: str
    search_query: str
    search_results: str
    draft_report: str
    messages: Annotated[list, operator.add]
```

**完整实例项目：**
```python
class InformationRetrievalState(TypedDict):
    topic_to_analyze: str
    generated_query: str
    thinking_result: dict
    search_results: str

class ReportGenerationState(TypedDict):
    search_context: str
    original_topic: str
    draftwriter_thinking_result: dict
    reviewerllm_thinking_result: dict
    first_draft: str
    draft_content: str
    polished_report: str
    final_document: str

class AgentState(TypedDict):
    topic: str
    search_query: str
    draft_report: str
    messages: Annotated[list, operator.add]
    llm_messages: Annotated[list, operator.add]
    informationretrieval: InformationRetrievalState
    reportgeneration: ReportGenerationState
```

**差异说明：**
1. 完整项目将状态拆分为子状态结构（`InformationRetrievalState`、`ReportGenerationState`），便于模块隔离管理
2. 新增 `llm_messages` 字段用于存储仅包含 system prompt 和 user prompt 的消息
3. 模块状态字段使用 `Annotated[list, operator.add]` 注解，支持增量更新

---

### 3.2 build.py 差异

**空壳项目特征：**
- 简单的 `workflow.add_edge()` 和 `workflow.add_conditional_edges()`
- 条件边函数返回固定目标或 TODO 占位符
- 未实现 ToolMessage 提取逻辑

**完整实例项目特征：**
- 每个 LLM 节点后添加对应的 `_response` 工具节点（如 `think_generatequery_response`、`generatequery_response`）
- 条件边函数从 `state["messages"]` 中提取 `ToolMessage`
- 提取的 ToolMessage 内容存入模块专用状态（如 `state["informationretrieval"]["thinking_result"]`）

**示例 - route_after_think_generatequery_response：**
```python
def route_after_think_generatequery_response(state: AgentState) -> str:
    if "informationretrieval" not in state:
        state["informationretrieval"] = {}
    
    for msg in state["messages"]:
        if hasattr(msg, "tool_call_id") and hasattr(msg, "content"):
            tool_name = getattr(msg, "name", None)
            tool_content = msg.content
            
            if tool_name == "think_generatequery_response":
                state["informationretrieval"]["thinking_result"] = tool_content
    
    return "generatequery"
```

### 3.3 models.py 差异

**空壳项目：**
- 思考模型使用 `pass` 占位，无结构化字段
- 响应模型简单定义输出字段

**完整实例项目：**
- 思考模型定义完整的结构化字段，用于引导 LLM 进行思维链推理
- 字段带有详细的 `description`，帮助 LLM 理解输出要求

**示例 - think_generatequery_model：**
```python
class think_generatequery_model(BaseModel):
    analysis: str = Field(description="对主题的分析和理解")
    keyword_extraction: List[str] = Field(description="从主题中提取的潜在关键词")
    search_strategy: str = Field(description="搜索策略和方法")
```

### 3.4 prompts.py 差异

**空壳项目：**
- 系统提示词简单、通用
- 缺少具体的任务指令和工具调用指导

**完整实例项目：**
- 详细的系统提示词，包含任务要求、输入参数说明、输出格式要求
- 明确要求 LLM 使用工具调用（如 "**必须调用 websearchtool 工具执行搜索**"）
- 包含分步骤的思考指导

**示例 - generatequery 提示词：**
```python
"generatequery": """你是一个智能助手，负责从用户提供的主题中提取最适合Google搜索的关键词，并执行搜索。
```

### 3.5 节点实现差异

**空壳项目节点（generatequery.py）：**
```python
def generatequery(state: AgentState) -> AgentState:
    llm = get_llm()
    from .tools import TOOLS
    llm_with_tools = llm.bind_tools(TOOLS)
    
    system_prompt = get_system_prompt("generatequery")
    messages = [SystemMessage(content=system_prompt)] + state["messages"]
    
    response = llm_with_tools.invoke(messages)
    state["messages"].append(response)  # 直接修改原状态
    
    return state
```

**完整实例项目节点：**
```python
def generatequery(state: AgentState) -> AgentState:
    if "informationretrieval" not in state:
        state["informationretrieval"] = {}
    
    llm = get_llm()
    from .tools import websearchtool
    llm_with_tools = llm.bind_tools([websearchtool])  # 只绑定需要的工具
    
    system_prompt = get_system_prompt("generatequery")
    topic = state["informationretrieval"].get("topic_to_analyze", "")
    thinking_result = state["informationretrieval"].get("thinking_result", {})
    
    # 构建 llm_messages
    llm_messages = [
        SystemMessage(content=system_prompt),
        HumanMessage(content=f"主题: {topic}")
    ]
    if thinking_result:
        llm_messages.append(HumanMessage(content=f"思考结果: {thinking_result}"))
    
    response = llm_with_tools.invoke(llm_messages)
    
    # 提取生成的查询词并存储到模块状态
    if hasattr(response, "content") and response.content:
        generated_query = response.content.strip()
    
    # 返回增量状态更新
    return {"messages": [response], "informationretrieval": {"generated_query": generated_query}}
```

**关键差异：**
1. 节点函数不再直接修改 `state`，而是返回增量更新字典
2. 从模块专用状态中读取输入（如 `state["informationretrieval"]["topic_to_analyze"]`）
3. 将输出写入模块专用状态（如 `state["informationretrieval"]["generated_query"]`）
4. 只绑定需要的工具，而非全部工具

### 3.6 tools.py 差异

**空壳项目：**
- 工具函数返回固定字符串 `"Response recorded"` 或 `"Thinking completed"`
- 无实际业务逻辑

**完整实例项目：**
- 响应工具返回原始输入内容（如 `return first_draft`）
- 思考工具返回结构化的思考结果
- 实现了实际功能（如 `websearchtool` 实现了 Bing 搜索功能）

**示例 - websearchtool 实现：**
```python
@tool(args_schema=websearchtool_model)
def websearchtool(query_string: str) -> str:
    try:
        import requests
        from bs4 import BeautifulSoup
        
        url = f"https://www.bing.com/search?q={query_string}&count=10"
        headers = {"User-Agent": "Mozilla/5.0 ..."}
        response = requests.get(url, headers=headers, timeout=10)
        
        # 解析搜索结果...
        result = f"搜索结果 for '{query_string}':\n"
        # ...
        return result
    except Exception as e:
        return f"搜索工具执行出错: {str(e)}"
```

### 3.7 main.py 差异

**空壳项目：**
- 条件边函数 `ot2` 返回固定的 `"target_node"` 占位符
- 初始状态仅包含基本字段
- 未实现模块间数据传递

**完整实例项目：**
- 条件边函数检查 `state["informationretrieval"]["search_results"]` 是否存在
- 将搜索结果传递给 `reportgeneration` 模块
- 完整的初始状态定义，包含所有模块状态

**示例 - ot2 条件边函数：**
```python
def ot2(state: AgentState) -> str:
    if "informationretrieval" in state and "search_results" in state["informationretrieval"] and state["informationretrieval"]["search_results"]:
        if "reportgeneration" not in state:
            state["reportgeneration"] = {}
        state["reportgeneration"]["search_context"] = state["informationretrieval"]["search_results"]
        state["reportgeneration"]["original_topic"] = state["informationretrieval"].get("topic_to_analyze", state.get("topic", ""))
        return "reportgeneration"
    else:
        return END
```

## 四、LGBS 模板改进建议

### 4.1 状态管理改进

1. **采用模块化状态结构**
   - 在 `state.py` 中预定义各模块的子状态类型
   - 提供状态初始化模板，确保每个模块状态包含必要字段

2. **分离消息管理**
   - 保留 `messages` 用于完整消息历史
   - 新增 `llm_messages` 用于 LLM 调用隔离

### 4.2 工作流架构规范

1. **Tool Calling 完整闭环**
   - 每个 LLM 节点后必须接 `_response` 工具节点
   - 条件边函数必须从 messages 中提取 ToolMessage 并存入状态

2. **模块间数据传递**
   - 预定义模块间传递的状态字段
   - 在 `main.py` 条件边函数中实现完整的数据传递逻辑

### 4.3 提示词模板

1. **详细的系统提示词**
   - 包含任务要求、输入参数、输出格式
   - 明确指导 LLM 使用工具调用

2. **思考节点提示词**
   - 提供分步骤的思考指导
   - 要求输出结构化的思考结果

### 4.4 工具实现

1. **响应工具**
   - 返回原始输入内容，而非固定字符串
   - 确保 LLM 输出的结构化数据能被正确传递

2. **思考工具**
   - 定义完整的 Pydantic 模型字段
   - 在函数实现中返回结构化的思考结果

## 五、开发指导总结

用户基于 LGBS 空壳项目开发时，应按以下步骤填充内容：

1. **完善 state.py**
   - 定义模块子状态类型
   - 添加必要的状态字段

2. **实现 tools.py**
   - 编写实际的工具函数逻辑
   - 确保响应工具返回正确的值

3. **填充 prompts.py**
   - 编写详细的系统提示词
   - 指导 LLM 正确使用工具

4. **完善 models.py**
   - 定义思考模型的结构化字段
   - 添加详细的 Field description

5. **更新 build.py**
   - 实现条件边函数中的状态提取逻辑
   - 确保 ToolMessage 被正确处理

6. **修改节点实现**
   - 从模块状态读取输入
   - 将输出写入模块状态
   - 返回增量状态更新

7. **完善 main.py**
   - 实现模块间的条件边函数
   - 传递必要的数据
