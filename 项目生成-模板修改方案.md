# LGBS 项目生成-模板修改方案

## 一、项目背景与目标

### 1.1 背景

LGBS 项目目前可以根据可视化配置自动部署一个空壳 LangGraph 项目，但生成的代码缺少完整的实现细节，需要用户手动填充大量内容。通过分析 `DeepResearchAgent.json` 参考文件和 `差异研究报告.md`，我们发现了当前模板与完整实例项目之间的关键差异。

### 1.2 目标

- **最大化自动生成**：根据 `DeepResearchAgent.json` 参考文件，尽可能生成完整的可执行代码
- **最小化手动干预**：对于无法自动生成的内容，提供详细的开发指导
- **标准化架构**：采用模块化、结构化的设计，确保生成的代码具有一致性和可维护性

---

## 二、当前模板分析

### 2.1 现有文件结构

```
backend/src/api/endpoints/
├── codegen.py        # 代码生成逻辑
├── templates.py      # 代码模板定义
└── project.py        # 项目管理接口
```

### 2.2 现有模板问题

1. **状态管理**：扁平化状态结构，缺少模块化状态管理
2. **工作流架构**：简单的节点连接，缺少完整的 Tool Calling 闭环
3. **节点实现**：直接修改 state，缺少增量更新和模块状态读写
4. **工具实现**：返回固定字符串，缺少实际业务逻辑
5. **模块间传递**：无数据传递逻辑，需要手动实现
6. **提示词**：通用提示词，缺少具体的任务指令

---

## 三、模板修改计划

### 3.1 核心文件修改

#### 3.1.1 state.py 模板修改 

**修改目标**：实现模块化状态结构

**修改内容**：
- 为每个模块生成子状态类型（如 `InformationRetrievalState`、`ReportGenerationState`）
- 在 `AgentState` 中添加模块状态字段
- 添加 `llm_messages` 字段用于 LLM 调用隔离
- 确保所有状态字段使用正确的类型注解

**修改文件**：`backend/src/api/endpoints/codegen.py` 中的 `generate_state` 函数

#### 3.1.2 build.py 模板修改

**修改目标**：实现完整的 Tool Calling 闭环

**修改内容**：
- 为每个 LLM 节点生成对应的 `_response` 工具节点
- 实现 `route_after_*` 条件边函数，提取 ToolMessage 并存入模块状态 # 建议: 这里注意只需要给出函数即可,函数内部不需要实现,但是你可以写个注释提醒用户怎么改
- 支持模块间的条件边逻辑

**修改文件**：
- `backend/src/api/endpoints/codegen.py` 中的 `generate_module_build` 函数
- `backend/src/api/endpoints/templates.py` 中的 `MODULE_BUILD_TEMPLATE` 和 `THINK_CONDITION_TEMPLATE`

#### 3.1.3 节点模板修改

**修改目标**：支持模块状态读写和增量更新

**修改内容**：
- 节点函数返回增量更新字典，而非直接修改 state
- 从模块专用状态中读取输入（如 `state["informationretrieval"]["topic_to_analyze"]`）
- 将输出写入模块专用状态
- 只绑定需要的工具，而非全部工具 # 建议: 模板里用全部工具是因为我们写模板的时候确定不了哪个llm要用哪个工具,所以这里也是绑定全部或者随便设置几个占位符作用的变量,如t1,t2,t3,然后在注释里告诉用户到时候自己写的时候要换成llm节点对应的工具

**修改文件**：
- `backend/src/api/endpoints/codegen.py` 中的 `generate_node_file` 函数
- `backend/src/api/endpoints/templates.py` 中的 `NODE_FILE_TOOL_CALLS_TEMPLATE`

#### 3.1.4 tools.py 模板修改

**修改目标**：生成更完整的工具实现

**修改内容**：
- 响应工具返回原始输入内容，而非固定字符串 #建议,这里,我觉得你这种不好搞,因为工具太灵活了,模板也兼顾不了大部分情况,所以这里直接返回一个result对象然后注释写清楚result可以是xxxxx类型,用户自己根据情况写
- 思考工具返回结构化的思考结果
- 为常见工具（如 websearch）提供基础实现

**修改文件**：
- `backend/src/api/endpoints/codegen.py` 中的 `generate_module_tools` 函数
- `backend/src/api/endpoints/templates.py` 中的 `CUSTOM_TOOL_TEMPLATE` 和 `THINK_TOOL_TEMPLATE`

#### 3.1.5 main.py 模板修改

**修改目标**：支持模块间数据传递

**修改内容**：
- 实现模块间的条件边函数，传递必要的数据 # 建议:这里你可以给模块加上通信函数,因为节点和模块都是确定的,但是函数体还是置空,在注释里告诉我用户要通过状态来存储数据,比如把想存的数据存到state的某个属性中
- 生成完整的初始状态，包含所有模块状态 # 建议:咱们写模板搞不出全部状态,这里也在state的模板那里写清楚注释告诉用户在这里加上开发时需要的状态
- 提供详细的结果输出逻辑 

**修改文件**：
- `backend/src/api/endpoints/codegen.py` 中的 `generate_main` 函数
- `backend/src/api/endpoints/templates.py` 中的 `MAIN_TEMPLATE`

#### 3.1.6 prompts.py 模板修改

**修改目标**：生成更详细的系统提示词

**修改内容**：
- 包含任务要求、输入参数说明、输出格式要求 #这里你还是把这些写到注释里让用户决定咋写
- 明确指导 LLM 使用工具调用 # 建议: 这里你可以写个注释告诉用户,比如"你可以用think工具来思考,用response工具来响应",然后用户根据情况自己写
- 为思考节点提供分步骤的思考指导 # 建议: 这里你可以写个注释告诉用户,比如"你可以先思考一下,然后用think工具来思考,用response工具来响应",然后用户根据情况自己写


**修改文件**：
- `backend/src/api/endpoints/codegen.py` 中的 `generate_module_prompts` 函数
- `backend/src/api/endpoints/templates.py` 中的 `PROMPTS_TEMPLATE`

#### 3.1.7 models.py 模板修改

**修改目标**：生成更完整的 Pydantic 模型

**修改内容**：
- 为思考模型定义完整的结构化字段 # 建议: 维持现在的pass不变,写一个注释告诉用户这里要写完整的结构化字段,至于要哪些字段,用户根据节点需要生成啥来写
- 字段带有详细的 `description`，帮助 LLM 理解输出要求 # 建议:也是写到注释里
- 支持复杂的数据结构（如 List[str]） # 建议:也是写到注释里

**修改文件**：
- `backend/src/api/endpoints/codegen.py` 中的 `generate_module_models` 函数
- `backend/src/api/endpoints/templates.py` 中的 `THINK_MODEL_TEMPLATE`

#### 3.1.8 README.md 模板修改 

**修改目标**：提供详细的开发指导 # 建议: 你先别写这个,等你把所有模板中该加的注释都加完了,把这些注释整合一下放到这里来教用户就行,对了还得有一些部署,配置的说明比如如何启动,如何添加env变量,如何配置llm等

**修改内容**：
- 添加详细的项目结构说明
- 提供工具安装和配置指南
- 详细说明如何填充和修改生成的代码
- 提供常见问题解决方案

**修改文件**：
- `backend/src/api/endpoints/codegen.py` 中的 `generate_readme` 函数
- `backend/src/api/endpoints/templates.py` 中的 `README_TEMPLATE`

### 3.2 配置文件修改

#### 3.2.1 config.py 模板修改

## 四、实现策略

### 4.1 数据结构映射

根据 `DeepResearchAgent.json` 参考文件，实现以下映射：

1. **全局状态** → `AgentState` 中的基本字段
2. **模块定义** → 生成对应的模块子状态类型
3. **节点定义** → 生成节点文件和思考节点
4. **连接定义** → 生成条件边函数和路由逻辑

### 4.2 代码生成逻辑

1. **状态生成**：根据模块定义生成子状态类型
2. **工具生成**：根据工具节点定义生成工具函数和模型
3. **节点生成**：生成 LLM 节点和思考节点
4. **构建生成**：生成包含思考节点和工具节点的工作流
5. **主文件生成**：生成包含模块间传递逻辑的主文件

### 4.3 智能默认值

- **系统提示词**：根据节点类型和描述生成默认提示词
- **工具实现**：为常见工具（如 websearch）提供基础实现
- **状态管理**：自动处理模块状态的初始化和更新
